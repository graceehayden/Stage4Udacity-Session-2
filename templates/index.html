Lesson 5: Debugging

Bugs Happen!

Problems within code that prevent it from working properly are referred to as "bugs". They can't be avoided, so it is important to build a mental model and techniques that help to find out how the system works and what about it isn't.

Error Messages

Error messages are provided when code crashes. It looks confusing, but does usually include useful information about what type of error there is and where. Solutions to error messages can often be found through Google.

Work Off of Example Code

Code is often written using example code that works properly. You can work off of the example and change it to do what you want it to.

If that doesn't work, make random changes and, if still struggling, ask for help. After that, try commenting out the defunct new code and copying the example code. Run the example to make sure it works. Then, modify it in small steps to make sure it does what you want. Once you have functional code, look back and try to see where you went wrong before.

If working with example code, you must take care to make sure the example itself works properly or the new code may inherit errors.

Check Intermediate Results

When dysfuntional code does not crash and get an error message, it can be harder to figure out what is going wrong. One way to go about working in this scenario is splitting up the code by introducing print statements that output the variables so you can see them. This helps to pinpoint where the function is going wrong.

Keep and Compare Old Versions

When code isn't working, it's tempting to throw it away. However, the first attempt may turn out to be better than the second and you may wish to go back to it. To save previous code, comment it out by adding a hashtag or turn it into a big string by adding triple quotes at either end of it. You may also copy code into a different file.

Software engineers use version control as well, such as Github, which allows them to access all versions worked with up to the current version.

Lesson 3: Use Classes

What is a Class?

An object class is like a blueprint of a building. The same blueprint can be used to build multiple buildings with different characteristics, such as the color or how many stories it has. That class can be used to create multiple instances of that blueprint by building off of its existing properties as a new object.

For instance, within the class "Dog", there could be many instances, such as "boxer", "bulldog", etcetera. Each of those instances would have the same properties that all dogs do, but they would also have some that make them unique.

Using Abstracted Code from Modules

"from" and "import" statements allow you to import specific attributes from a module. When you download a package file or module, you get code that is essentially hidden, or abstracted away, but will perform its job nonetheless.

There are classes within those modules that, when called as a function with parentheses, in turn call the __init__ function defined inside of that class. This creates space and memory for a new instance, also called an object. It will have its own unique attributes, or properties.

Make Classes

Google style guide provides guidelines for styling conventions when naming new classes. It is good practice to create a new class inside one file and then put it to use in another.

First, create a class outside of the Python Standard Library. Then subclasses can be created within it. This calls the function __init__, also known as a constructor, because it constructs space or memory for a new instance.

The __init__ function is a preexisting class variable. It takes a few parameters, the first of which is self. Self is the name of the instance itself. The other variables are the information about it and are called instance variables. Every object has its own version of those variables.

Class variables, in contrast, are variables that we want all the objects in that class to share. For instance, for a class Movie, valid_ratings would be a variable that applied to all instances because all movies are rated on the same scale.

Inheritance, as in genetics, is the passing down of attributes from one class to another, newer class. To use those attributes, the parent class's __init__ function must be called within the child class's __init__ function. It must also be referenced in the class parenthesis --> class Child(Parent)

An inherited method can be overridden with a method of the same name within a subclass. This is called method overriding.

Abstraction is applied in creating new classes and objects by the inheritance and inclusion of code that is written elsewhere. Similar to HTML and CSS, where the CSS styling is done externally and imported into the HTML document for simplicity's sake, so it goes with programming. This is abstraction of unnecessary information.

Methods are functions that take an instance as the first parameter and defines some characteristics of a class/object. They are attributes of classes or objects, which differs from a function in that a function does NOT belong to a class. The object (self) is passed in as follows: self.name = name . A method for class dog, for example, could be a dog eats or walks.

Object-oriented Programming (OOP)

Now, after explaining classes and objects, object-oriented programming is a programming model in which objects are created to interact in a web of data, each containing its own data.

With a main idea class defined, like fruit, you can then break the smaller parts into types of fruit which have their own unique properties in addition to those that all fruits share.

Functions that don't need to be seen are abstracted away. If I make a cake, I don't need to know how the ingredients came to be in order to cook with them. Similarly, OOP utilizes objects and their functions without always having to tie in extraneous information.

What is the Importance of OOP?

OOP makes programming for large projects much simpler and more streamlined because the path to the end product can be mapped out like a family tree, with variables being passed down rather than being created over and over again.

A defined class can be used many times over in the program it was created for, but also by other OO programs. This lends itself to easier distribution within networks.

Any data type can be created with this class/object creation method, which means the possibilities are endless!

Lesson 1 - Making A Web Application

Networking, the Internet, and Servers

A network is a group of entities (communities, governments, computers) that can communicate even though they are not directly connected. For example, a<->b<->c would be a network because "a" cannot communicate directly with "c", but can do so through "b". There must be at least 3 nodes in a network. To make a network, we need a way to encode messages, a way to interpret them, a way to route them, and rules for who gets to use the resources.

There are two main ways to measure a network: latency and bandwidth. Latency is the time it takes for a message to get from source to destination. We measure this by timing when you start sending to the time that the receiver starts receiving. It is a unit of time often measured in milliseconds. Bandwidth is the amount of information that can be transmitted per unit time. It is measured in bits per second, or Mbps(million bits per sec).

What is a bit? One bit is the smallest unit of information; it can transmit one yes or no, or a zero or one - only two choices. The more bits you have, the more information you can encode. For example, 4 bits can encode 16 different numbers. Every time I add one more bit, I double the information because I have one more yes/no question.

Networks need protocols. Protocol is a set of rules for how two entities can talk to each other. On the web, we use hypertext transfer protocol (http). This is how a client (the web browser) can talk to the server to obtain a document. The client can send a GET message to receive something and then the server sends a response.

What is going on in that URL?

URL stands for uniform resource locator. For example, in http://www.udacity.com/ --> "http" is the protocol, "udacity" is the host, and the forward slash plus anything after it is the path.

A query parameter(or GET parameter) will be within the path, starting with a question mark; the structure of it will be name=value. A second parameter and any following parameters, will be separated by ampersands.

For example: http://udacity.com/foo?p=foo -> The value of p is foo.

A fragment is something following a pound sign in the path of a URL. It is not sent to the server when you make a request, but it is used to reference a particular part of a page that you're looking at. When there are query parameters in the URL, the fragment follows last.

The final piece to a URL is the port. In order to make an internet connection, you need the address and the port. By default, the port equals 80. If you want a different one, you can include it in the URL between the host and the path. For example, http://localhost:8000/

HTTP, again, is what the browser uses to talk to the web server. Requests from your browser (http://www.example.com/foo) begin with a request line. The request line has 3 main parts: the method, path, and version.

Example -> GET /foo HTTP/1.1

GET is the method, /foo is the path, and HTTP/1.1 is the version. The method is the type of request being made (ie. get, post). The path is the actual document we're requesting from the server. The version is most frequently 1.1. The host isn't in the request line. It is only used for making the connection. The path is used for making the request.

Headers for HTTP Requests include Host and User-agent. For example, (Host: www.example.com) and (User - Agent: chrome). User-agent is usually the type of browser. User agents are a very important part of headers. It is important to include it honestly when writing software that interacts with others' websites. The format in writing a user agent is no space between words in the name + a space + a colon + a space + the value.

Example: Host: www.hipmunk.com or User-Agent: ignore me.

HTTP Responses are what the server sends back to the requesting browser. The response looks very similar to the request. It has a status line, which is analogous to the request line.

For example: ( - HTTP/1.1 200 OK) --> "200" is the status code, "OK" is the reason phrase, and the version is the same. The reason phrase is the English language description of the status code: "200 OK" means the document is found, "302 found" means the document is located somewhere else, "404 Not Found" means that the document wasn't found, and "500 Server Error" means the server broke trying to handle the request. Common headers for HTML responses include Date, Server (ie. Server: Apache/2.2.3), Content-Type, and Content-Length.

SERVERS: the purpose of a server is to respond to HTTP requests. The two main types of responses are static and dynamic. Content is considered static if it is a pre-written document that the server just returns. An image would be an example. Dynamic requests are where the response is built on the fly. This content generation is done by web applications. A web application lives on a web server, it speaks HTTP, and it generates content that the browser requests.

Databases

Databases are how we store all the information being collected by users and are the final piece in creating functional web applications.

Google App Engine's data store can abstract away a lot of the difficulty in creating a database, which makes the creation of a functional web application a lot simpler.

A database is a program that stores and retrieves large amounts of structured data (see "Tables" below). This can be just that program, the machine running that program, or a group of machines working together to store/retrieve data.

Tables are used pretty universally in databases. For example, a link submitted to Reddit is broken up into columns, such as ID, votes, user who submitted the link, the date it was submitted, the display title, and the URL. Each new entry would be a new row in the table. One row will be one instance of an element in the table. The user column may come from another table of users, which may have ID, name, date, and password columns.

Querying a database using Python code is a difficult task. It is error-prone, tedious, and slow.

There are many types of databases. Relational databases often use SQL to manipulate them. Examples of this type include MySQL, PostgreSQL, SQLite, and Oracle. Google App Engine's Data Store and Amazon Dynamo are other types of databases. There are also NoSQL databases, which include Mongo and Couch as popular options.

SQL, or sequel, or structured query language, is a language for expressing queries and asking question of the database.
